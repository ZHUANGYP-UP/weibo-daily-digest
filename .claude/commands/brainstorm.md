# 需求探索与设计

通过自然协作对话，将想法转化为完整的设计文档和规格说明。

先理解当前项目上下文，然后逐一提问来细化想法。理解清楚后，展示设计方案并获得用户批准。

<HARD-GATE>
在展示设计方案并获得用户批准之前，绝不编写任何代码、搭建项目或执行任何实现操作。这适用于每个项目，无论看起来多简单。
</HARD-GATE>

## 反模式："这太简单了不需要设计"

每个项目都必须经过这个流程。待办清单、单函数工具、配置修改——全都包括。"简单"项目恰恰是未审视的假设造成最多浪费的地方。设计可以很简短（真正简单的项目几句话即可），但必须展示并获得批准。

## 检查清单

必须为以下每项创建任务并按顺序完成：

1. **探索项目上下文** — 检查文件、文档、最近提交
2. **提出澄清问题** — 一次一个，理解目的/约束/成功标准
3. **提出 2-3 个方案** — 附带权衡分析和推荐
4. **展示设计** — 按复杂度分段展示，每段获得用户批准
5. **撰写设计文档** — 保存到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 并提交
6. **规格自查** — 快速检查占位符、矛盾、歧义、范围
7. **用户审阅文档** — 请用户在继续之前审阅规格文件

## 流程

1. 查看当前项目状态（文件、文档、最近提交）
2. 评估范围：如果需求描述了多个独立子系统，立即标记并协助拆解
3. 逐一提问来细化想法
4. 提出 2-3 个方案及权衡分析
5. 逐段展示设计方案
6. 撰写规格文档到 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
7. 自查占位符、矛盾、歧义
8. 请用户审阅写好的规格文档

## 核心原则

- **一次一个问题** — 不要用多个问题淹没用户
- **优先多选题** — 比开放题更容易回答
- **严格执行 YAGNI** — 从所有设计中移除不必要的功能
- **探索替代方案** — 确定方案前始终提出 2-3 个选项
- **增量验证** — 展示设计，获得批准后再继续

## 隔离与清晰的设计

- 将系统拆分为职责单一的更小单元
- 对每个单元回答：它做什么、怎么用、依赖什么？
- 不看内部实现就能理解一个单元的职责吗？
- 遵循代码库中的现有模式

## 规格自查

写完规格文档后：
1. **占位符扫描** — 有没有 "TBD"、"TODO"、不完整的章节？修复它们。
2. **内部一致性** — 各章节之间有没有矛盾？
3. **范围检查** — 聚焦度是否足以支撑单个实现计划？
4. **歧义检查** — 有没有需求可以有两种理解方式？选定一种并明确写出。

发现的问题直接修复，不需要重新审阅——修完继续。

## 规格文档评审

如果需要第二遍检查，派发评审子代理：

```
Agent(
  subagent_type: "general-purpose",
  description: "评审规格文档",
  prompt: "你是规格文档评审者。验证此规格是否完整且可用于制定计划。\n\n**待评审规格：** [SPEC_FILE_PATH]\n\n## 检查内容\n\n| 类别 | 关注点 |\n|------|--------|\n| 完整性 | TODO、占位符、TBD、不完整章节 |\n| 一致性 | 内部矛盾、冲突的需求 |\n| 清晰度 | 足够模糊以致可能构建错误功能的需求 |\n| 范围 | 是否聚焦于单个计划 |\n| YAGNI | 未要求的功能、过度设计 |\n\n只标记会在实现计划阶段造成实际问题的情况。除非有严重缺陷，否则批准。"
)
```
