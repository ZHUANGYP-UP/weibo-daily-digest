# Git Commit 约定式提交

使用约定式提交规范创建标准化、语义化的 git 提交。分析实际 diff 来确定合适的类型、范围和消息。

## 约定式提交格式

```
<类型>[可选范围]: <描述>

[可选正文]

[可选脚注]
```

## 提交类型

| 类型       | 用途                           |
| ---------- | ------------------------------ |
| `feat`     | 新功能                         |
| `fix`      | Bug 修复                       |
| `docs`     | 仅文档变更                     |
| `style`    | 格式/样式（无逻辑变更）        |
| `refactor` | 代码重构（非功能/修复）        |
| `perf`     | 性能优化                       |
| `test`     | 添加/更新测试                  |
| `build`    | 构建系统/依赖变更              |
| `ci`       | CI/配置变更                    |
| `chore`    | 维护/杂项                      |
| `revert`   | 回退提交                       |

## 工作流程

### 1. 分析 Diff

```bash
git diff --staged    # 暂存区 diff
git diff             # 工作区 diff
git status --porcelain
```

### 2. 暂存文件（如需要）

按逻辑分组暂存，**永远不要提交密钥**（.env、credentials.json、私钥）。

### 3. 生成提交消息

分析 diff 确定类型、范围和描述（现在时态、祈使语气、<72 字符）。

### 4. 执行提交

```bash
git commit -m "$(cat <<'EOF'
<类型>[范围]: <描述>

<可选正文>
EOF
)"
```

## 最佳实践

- 每次提交一个逻辑变更
- 使用现在时态："add" 而非 "added"
- 使用祈使语气："fix bug" 而非 "fixes bug"
- 引用 Issue：`Closes #123`、`Refs #456`
- 描述保持在 72 字符以内

## Git 安全协议

- 绝不修改 git 配置
- 未经明确请求，绝不运行破坏性命令（--force、hard reset）
- 除非用户要求，绝不跳过钩子（--no-verify）
- 绝不强制推送到 main/master
- 如果提交因钩子失败，修复后创建新提交（不要 amend）
